# Model Cards for Model Reporting — Mitchell et al.

## 결론부터

이 논문의 결론은 이겁니다.

**머신러닝 모델도 “모델 설명서”를 가져야 한다.**  
모델이 어떤 목적으로 만들어졌는지, 어디에 써도 되는지, 어디에 쓰면 안 되는지, 어떤 데이터로 평가했는지, 어떤 집단이나 조건에서 성능이 나빠지는지, 어떤 윤리적 위험이 있는지를 짧고 표준화된 문서로 공개해야 한다는 주장입니다.

저자들은 이 문서를 **Model Card**라고 부릅니다. Model card는 학습된 머신러닝 모델에 딸려 나오는 짧은 문서로, 모델의 의도된 사용 맥락, 평가 절차, 성능 특성, 집단별·조건별 평가 결과를 담아야 합니다.

---

## 이 논문을 한 문장으로 요약하면

**모델의 평균 성능만 보고 “좋은 모델”이라고 말하지 말고, 어떤 상황과 어떤 사람들에게 잘 작동하는지까지 보고하라.**

예를 들어 어떤 얼굴 인식 모델의 전체 정확도가 95%라고 해봅시다.

겉으로는 좋아 보입니다.

하지만 집단별로 나누어 보면 이런 결과일 수 있습니다.

| 집단 | 정확도 |
|---|---:|
| 밝은 조명, 정면 얼굴 | 99% |
| 어두운 조명, 측면 얼굴 | 82% |
| 젊은 성인 | 97% |
| 고령층 | 78% |
| 특정 피부톤 집단 | 70% |

전체 정확도 95%는 이런 차이를 숨길 수 있습니다.

이 논문이 말하는 핵심은 바로 이것입니다.

> **평균 성능은 모델의 실패를 감출 수 있다.  
> 그래서 모델 성능은 사용 맥락별, 집단별, 조건별로 쪼개서 보고해야 한다.**

## Datasheets for Datasets와의 관계

이전 논문인 **Datasheets for Datasets**가 데이터셋의 설명서라면, 이번 논문인 **Model Cards for Model Reporting**은 모델의 설명서입니다.

| 문서 | 설명 대상 | 핵심 질문 |
|---|---|---|
| **Datasheet** | 데이터셋 | 이 데이터는 어디서 왔고, 어떻게 수집됐고, 어디에 써도 되는가? |
| **Model Card** | 학습된 모델 | 이 모델은 어떤 용도로 만들었고, 어떤 조건에서 잘하거나 못하는가? |

즉, 둘을 같이 쓰면 이런 구조가 됩니다.

```text
Datasheet: 데이터의 출처와 한계
        ↓
Model Card: 그 데이터로 학습한 모델의 성능과 한계
        ↓
Deployment Decision: 이 모델을 실제로 써도 되는지 판단
```

---

## 왜 이 논문이 중요한가?

머신러닝을 처음 공부하면 보통 이렇게 생각합니다.

> “accuracy가 높으면 좋은 모델 아닌가?”

그런데 이 논문은 이렇게 묻습니다.

> “누구에게 높은 accuracy인가?”  
> “어떤 상황에서 높은 accuracy인가?”  
> “어떤 오류가 더 위험한가?”  
> “이 모델을 실제 서비스에 써도 되는가?”  
> “이 모델을 쓰면 안 되는 상황은 무엇인가?”

이 차이가 굉장히 중요합니다.

예를 들어 의료 진단 모델에서 false negative, 즉 실제로 병이 있는데 없다고 예측하는 오류는 매우 위험할 수 있습니다. 반대로 어떤 보안 시스템에서는 false positive, 즉 문제가 없는데 위험하다고 잘못 판단하는 오류가 사용자를 심각하게 불편하게 만들 수 있습니다.

## 핵심 개념 1: Model Card는 모델의 “영양성분표”에 가깝다

Model Card를 쉽게 이해하려면 식품 영양성분표를 떠올리면 됩니다.

음식을 살 때 우리는 단순히 “맛있다”만 보지 않습니다.

| 식품 설명서에서 보는 것 |
|---|
| 칼로리 |
| 단백질 |
| 지방 |
| 당류 |
| 나트륨 |
| 알레르기 유발 성분 |
| 유통기한 |
| 보관 방법 |

마찬가지로 모델도 단순히 “성능 좋음”이라고 써 있으면 부족합니다.

| Model Card에서 봐야 하는 것 |
|---|
| 모델명과 버전 |
| 개발자 또는 조직 |
| 모델 유형 |
| 의도된 사용처 |
| 사용하면 안 되는 상황 |
| 학습 데이터와 평가 데이터 |
| 성능 지표 |
| 집단별 성능 |
| 조건별 성능 |
| 윤리적 고려사항 |
| 한계와 권고사항 |

즉, Model Card는 모델을 배포하거나 재사용하려는 사람이 **이 모델을 믿고 써도 되는지 판단하도록 돕는 문서**입니다.

---

## 핵심 개념 2: 전체 성능보다 “쪼개진 성능”이 중요하다

이 논문에서 가장 중요한 표현 중 하나가 **disaggregated evaluation**입니다.

쉽게 말하면:

> 전체 평균 하나만 보지 말고, 성능을 집단별·조건별로 쪼개서 보라는 뜻입니다.

예를 들어 독성 댓글 탐지 모델이 있다고 해봅시다.

전체 AUC는 높을 수 있습니다. 하지만 특정 정체성 표현이 들어간 문장에서 성능이 크게 나빠질 수 있습니다.

| 문장 유형 | 모델 문제 가능성 |
|---|---|
| “I am gay.” | 독성이 없는데 toxic으로 잘못 판단 |
| “I am Muslim.” | 특정 집단 언급 자체를 위험 신호로 오해 |
| “I hate everyone.” | 실제 독성 문장을 놓칠 수 있음 |

이런 문제는 전체 accuracy나 AUC만 보면 잘 드러나지 않습니다.

모델 성능은 문화적, 인구통계적, 표현형적 집단뿐 아니라 age × race, sex × skin type 같은 교차 집단에 대해서도 평가해야 합니다.

---

## 핵심 개념 3: Intersectional Analysis

**Intersectional analysis**는 여러 특성이 겹친 집단을 따로 보는 분석입니다.

예를 들어 성별만 따로 보면 이렇게 볼 수 있습니다.

| 성별 | 오류율 |
|---|---:|
| 남성 | 5% |
| 여성 | 6% |

피부톤만 따로 보면 이렇게 볼 수 있습니다.

| 피부톤 | 오류율 |
|---|---:|
| 밝은 피부톤 | 4% |
| 어두운 피부톤 | 8% |

그런데 실제 문제는 교차 집단에서 나타날 수 있습니다.

| 성별 × 피부톤 | 오류율 |
|---|---:|
| 남성 × 밝은 피부톤 | 3% |
| 여성 × 밝은 피부톤 | 5% |
| 남성 × 어두운 피부톤 | 7% |
| 여성 × 어두운 피부톤 | 18% |

성별만 봤을 때도, 피부톤만 봤을 때도 “심각한 문제”가 덜 보였는데, 두 조건을 같이 보면 특정 집단에서 오류가 크게 높을 수 있습니다.

---

## 핵심 개념 4: “어디에 쓰면 안 되는가”를 반드시 써야 한다

Model Card에서 매우 중요한 항목은 **Intended Use**입니다.

그런데 intended use는 단순히 “어디에 써도 된다”만 쓰는 항목이 아닙니다.

반드시 **out-of-scope use cases**, 즉 사용하면 안 되는 상황을 써야 합니다.

예를 들어 미소 감지 모델이 있다고 합시다.

모델이 하는 일은 이미지에서 사람이 웃고 있는지를 판별하는 것입니다.

적절한 사용:

| 사용처 | 적절성 |
|---|---|
| 사진 앨범에서 웃는 사진 찾기 | 비교적 적절 |
| 재미용 필터 적용 | 비교적 적절 |
| 시각장애인을 위한 사진 설명 보조 | 신중히 사용 가능 |

부적절한 사용:

| 사용처 | 왜 부적절한가 |
|---|---|
| 사람의 감정 판단 | 웃는 표정과 실제 감정은 다름 |
| 면접 평가 | 위험하고 부정확한 추론 |
| 고객 만족도 자동 판단 | 표정만으로 만족도를 판단하기 어려움 |
| 정신건강 상태 추정 | 모델의 범위를 벗어남 |

많은 AI 사고는 모델이 나빠서만 생기는 것이 아니라, **모델을 원래 목적과 다른 곳에 가져다 쓰면서** 생깁니다.

## Model Card에 들어가는 주요 항목

| 항목 | 쉽게 말하면 | 대표 질문 |
|---|---|---|
| **Model Details** | 모델 기본 정보 | 누가 만들었나? 버전은? 모델 유형은? |
| **Intended Use** | 의도된 사용처 | 어디에 쓰라고 만든 모델인가? 어디에 쓰면 안 되는가? |
| **Factors** | 성능 차이를 만들 수 있는 요인 | 성별, 연령, 환경, 장비, 언어, 지역 등 어떤 요인이 중요한가? |
| **Metrics** | 평가 지표 | accuracy, FPR, FNR, AUC 등 무엇을 왜 썼는가? |
| **Evaluation Data** | 평가 데이터 | 어떤 데이터로 성능을 측정했는가? |
| **Training Data** | 학습 데이터 | 어떤 데이터로 학습했는가? 공개 가능한 정보는 무엇인가? |
| **Quantitative Analyses** | 정량 분석 | 집단별, 조건별, 교차 집단별 성능은 어떤가? |
| **Ethical Considerations** | 윤리적 고려 | 민감정보, 위험, 피해 가능성, 완화 전략은 무엇인가? |
| **Caveats and Recommendations** | 주의사항과 권고 | 어떤 한계가 있고, 추가로 무엇을 확인해야 하는가? |

이걸 한마디로 정리하면:

> Model Card는 모델의 “성능표 + 사용설명서 + 위험고지서”다.

## 핵심 개념 5: 지표는 사용 맥락에 맞게 골라야 한다

모델 평가에서 흔한 실수는 지표를 기계적으로 고르는 것입니다.

예를 들어 분류 문제라고 해서 무조건 accuracy만 보면 안 됩니다.

| 상황 | accuracy만 보면 안 되는 이유 |
|---|---|
| 사기 탐지 | 사기가 매우 드물어서 대부분 정상이라고 해도 accuracy가 높게 나올 수 있음 |
| 암 진단 | false negative가 매우 위험할 수 있음 |
| 스팸 필터 | false positive가 중요한 정상 메일을 막을 수 있음 |
| 혐오 발언 탐지 | 특정 집단 표현을 과도하게 toxic으로 판단할 수 있음 |
| 채용 모델 | 특정 집단의 false negative가 차별 문제로 이어질 수 있음 |

Model Card에서는 단순히 이렇게 쓰면 부족합니다.

> Accuracy: 92%

더 좋은 방식은 이런 식입니다.

| 항목 | 예시 |
|---|---|
| 주요 지표 | AUC, FPR, FNR, precision, recall |
| 기준 threshold | 0.5 |
| threshold 선택 이유 | false negative를 줄이는 방향으로 선택 |
| 전체 성능 | AUC 0.91 |
| 집단별 성능 | 연령대별, 성별별, 지역별로 보고 |
| 신뢰구간 | bootstrap 기반 95% CI |
| 중요 오류 | 고령층에서 FNR 높음 |

이렇게 해야 사용자가 모델의 실제 위험을 이해할 수 있습니다.

## 논문의 두 가지 예시

### 1. Smiling Classifier

첫 번째 예시는 이미지에서 미소를 감지하는 모델입니다.

논문은 CelebA 데이터셋으로 학습한 smiling classifier를 사용해 성별과 연령대별 성능을 보여줍니다. 이 분석에서 older men의 false discovery rate가 다른 집단보다 높고, 남성 집단 전체에서는 false negative rate가 높게 나타나는 등 전체 성능만으로는 보이지 않는 문제가 드러납니다.

| 발견 | 의미 |
|---|---|
| older men에서 FDR 높음 | 웃지 않았는데 웃었다고 잘못 판단하는 경우가 많음 |
| men 전체에서 FNR 높음 | 실제로 웃고 있는데 안 웃는다고 판단하는 경우가 많음 |
| race나 skin type 정보 없음 | 중요한 평가 축이 빠져 있음 |
| emotion detection에는 부적절 | 미소 표정과 감정은 다름 |

이 예시는 Model Card가 단순한 문서가 아니라, 모델의 약점을 드러내는 분석 도구라는 점을 보여줍니다.

### 2. Toxicity Scoring

두 번째 예시는 Perspective API의 독성 댓글 탐지 모델입니다.

이 모델은 인간 모더레이션 지원이나 댓글 작성자 피드백에는 쓰일 수 있지만 완전 자동 모더레이션이나 특정 개인에 대한 판단에는 의도된 것이 아니라고 적습니다.

예를 들어 초기 모델이 “gay”, “lesbian”, “homosexual” 같은 단어가 들어간 문장을 과도하게 toxic으로 판단했다면, 이는 단순 성능 문제가 아니라 특정 집단을 언급하는 정상 발화까지 억압하는 문제가 될 수 있습니다.

## 예시: 고객 이탈 예측 모델의 Model Card

고객 이탈 예측 모델을 만들었다고 해봅시다.

보통은 이렇게 보고합니다.

| 지표 | 값 |
|---|---:|
| AUC | 0.87 |
| Accuracy | 0.81 |
| F1-score | 0.62 |

하지만 Model Card 관점에서는 훨씬 더 많은 정보를 써야 합니다.

### 1. Model Details

| 항목 | 예시 |
|---|---|
| 모델명 | Subscription Churn Predictor v1.3 |
| 모델 유형 | XGBoost binary classifier |
| 개발 조직 | CRM 데이터사이언스팀 |
| 개발일 | 2026년 5월 |
| 문의처 | 데이터 플랫폼 담당자 |

### 2. Intended Use

| 항목 | 예시 |
|---|---|
| 권장 사용 | 이탈 가능성이 높은 고객을 선별해 retention 캠페인 우선순위화 |
| 권장 사용자 | CRM 마케팅팀, 고객 성공팀 |
| 비권장 사용 | 고객별 가격 차별, 신용평가, 계약 해지 자동 판단 |
| 주의 | 모델 점수는 “이탈 가능성”이지 “고객 가치”가 아님 |

### 3. Metrics and Segment Analysis

| 세그먼트 | AUC | Recall@Top10% |
|---|---:|---:|
| 전체 | 0.87 | 0.42 |
| 신규 고객 | 0.79 | 0.31 |
| 장기 고객 | 0.89 | 0.45 |
| 프리미엄 요금제 | 0.84 | 0.39 |
| 베이직 요금제 | 0.88 | 0.44 |

이런 식으로 쓰면 단순히 “AUC 0.87 모델”이 아니라, **어떤 고객군에서 불안정한지**가 보입니다.

## 입문자가 이 논문에서 꼭 배워야 할 점

### 1. 모델 성능은 하나의 숫자로 끝나지 않는다

`accuracy = 0.95`는 시작점일 뿐입니다.

그다음 질문이 중요합니다.

| 질문 |
|---|
| 어떤 데이터에서 0.95인가? |
| 어떤 집단에서 성능이 낮은가? |
| 어떤 오류가 더 위험한가? |
| threshold는 왜 그렇게 정했는가? |
| 실제 배포 환경과 평가 데이터가 비슷한가? |

### 2. 모델은 반드시 “사용 범위”를 가져야 한다

모델은 범용 마법 도구가 아닙니다.

모델은 특정 데이터, 특정 목적, 특정 평가 조건 안에서 만들어진 도구입니다.

따라서 좋은 모델 문서에는 반드시 이런 문장이 있어야 합니다.

> “이 모델은 A에는 사용할 수 있지만, B에는 사용하면 안 됩니다.”

### 3. 성능은 집단별로 확인해야 한다

특히 사람에게 영향을 주는 모델은 전체 평균만 보면 안 됩니다.

| 모델 | 꼭 확인해야 할 수 있는 집단별 성능 |
|---|---|
| 채용 모델 | 성별, 연령, 지역, 학력, 경력군 |
| 의료 모델 | 연령, 성별, 기저질환, 병원, 장비 |
| 신용 모델 | 소득구간, 지역, 직업군 |
| 음성인식 모델 | 언어, 억양, 성별, 연령, 소음 환경 |
| 이미지 모델 | 조명, 카메라, 피부톤, 촬영 각도 |

집단별 성능 차이는 공정성 문제일 수도 있고, 일반화 성능 문제일 수도 있고, 제품 안정성 문제일 수도 있습니다.

### 4. 평가 데이터 자체도 설명해야 한다

Model Card는 모델 성능만 적는 문서가 아닙니다. 성능을 측정한 **평가 데이터**가 무엇인지도 설명해야 합니다.

자율주행 모델을 평가할 때 맑은 날 낮 시간 데이터만 쓰면 부족합니다.

| 추가로 봐야 할 조건 |
|---|
| 야간 |
| 비 |
| 눈 |
| 안개 |
| 역광 |
| 공사 구간 |
| 보행자 밀집 지역 |
| 드문 교통 표지판 |

### 5. Model Card는 감사 보고서가 아니다

Model Card가 있다고 해서 모델이 안전하다는 뜻은 아닙니다.

> Model Card는 안전 보증서가 아니라, 투명성을 높이는 최소한의 보고 체계다.

제3자 감사, adversarial testing, 사용자 피드백, 배포 후 monitoring과 함께 쓰여야 합니다.

## 오늘 공부용 요약

**논문명:** Model Cards for Model Reporting  
**저자:** Margaret Mitchell, Simone Wu, Andrew Zaldivar, Parker Barnes, Lucy Vasserman, Ben Hutchinson, Elena Spitzer, Inioluwa Deborah Raji, Timnit Gebru  
**핵심 결론:** 학습된 머신러닝 모델에는 모델의 목적, 사용 범위, 평가 데이터, 성능 지표, 집단별 성능, 윤리적 고려사항, 한계와 권고사항을 담은 짧은 문서인 Model Card가 함께 제공되어야 한다.  
**왜 중요함:** 전체 평균 성능은 모델이 특정 집단이나 조건에서 실패하는 문제를 숨길 수 있다. Model Card는 모델 사용자가 그 모델을 어디에 써도 되는지, 어디에 쓰면 안 되는지 판단하도록 돕는다.  
**입문자가 배울 점:** 모델 성능은 하나의 accuracy로 끝나지 않는다. 평가 조건, 사용 맥락, 집단별 성능, 위험한 오류 유형, 비권장 사용처까지 함께 봐야 한다.  
**가장 중요한 문장:** 좋은 모델은 성능이 높은 모델만이 아니라, 자신의 사용 범위와 한계를 명확히 설명할 수 있는 모델이다.
