# Scaling Laws for Neural Language Models — Kaplan et al.

## 결론부터

이 논문의 결론은 이겁니다.

**언어 모델의 성능은 모델 크기, 데이터 크기, 학습 컴퓨트가 커질수록 꽤 예측 가능한 방식으로 좋아진다.** 특히 loss는 이 세 요소에 대해 대체로 **power law**, 즉 거듭제곱 법칙 형태로 감소합니다.

쉽게 말하면:

> “모델을 얼마나 크게 만들고, 데이터를 얼마나 많이 주고, 컴퓨트를 얼마나 쓰면 성능이 얼마나 좋아질지 어느 정도 예측할 수 있다.”

이 논문은 대규모 언어 모델을 만드는 일이 감이나 운에만 의존하는 것이 아니라, **스케일을 키웠을 때 성능이 어떻게 변하는지 경험적으로 예측 가능한 법칙이 있다**는 것을 보여줬습니다.

---

## 이 논문을 한 문장으로 요약하면

**Scaling Laws는 “LLM 성능은 모델 크기, 데이터, 컴퓨트를 키우면 예측 가능한 곡선을 따라 좋아진다”는 것을 실험적으로 보여준 논문입니다.**

GPT-3 논문과 연결하면 흐름이 명확합니다.

| 논문 | 핵심 질문 |
|---|---|
| **GPT-3** | 모델을 아주 크게 만들면 prompt만으로 여러 task를 풀 수 있는가? |
| **Scaling Laws** | 모델을 크게 만들 때 성능은 어떤 규칙으로 좋아지는가? |

즉, GPT-3가 “큰 모델이 few-shot learner가 될 수 있다”를 보여줬다면, Scaling Laws는 더 근본적으로 이렇게 묻습니다.

> 모델 크기를 키우면 성능이 얼마나 좋아질까?  
> 데이터가 부족하면 언제부터 성능이 막힐까?  
> compute 예산이 정해져 있다면 모델 크기와 학습량을 어떻게 나눠야 할까?

---

## 왜 이 논문이 중요한가?

이전까지는 큰 모델을 만들 때 이런 질문이 있었습니다.

```text
모델을 2배 키우면 성능이 얼마나 좋아질까?
데이터를 10배 늘리면 얼마나 좋아질까?
컴퓨트를 더 쓰면 어디까지 좋아질까?
```

이 논문은 이 질문에 대해 경험적 답을 줍니다.

```text
loss는 scale이 커질수록 부드럽고 예측 가능한 곡선을 따라 감소한다.
```

이게 중요한 이유는 LLM 개발이 매우 비싸기 때문입니다.

모델을 하나 학습하려면 GPU/TPU, 데이터, 시간, 엔지니어링 비용이 큽니다. 그래서 무작정 큰 모델을 학습하기 전에, 작은 모델들의 실험 결과로 큰 모델의 성능을 예측할 수 있다면 매우 유용합니다.

---

## 핵심 개념 1: Scaling Law

### 결론부터

**Scaling law는 어떤 시스템의 크기를 키웠을 때 성능이나 손실이 일정한 수학적 패턴을 따라 변하는 법칙입니다.**

이 논문에서는 주로 이런 관계를 봅니다.

```text
모델 크기 N이 커질수록 loss L이 어떻게 줄어드는가?
데이터 크기 D가 커질수록 loss L이 어떻게 줄어드는가?
컴퓨트 C가 커질수록 loss L이 어떻게 줄어드는가?
```

여기서 중요한 변수는 세 개입니다.

| 기호 | 의미 |
|---|---|
| `N` | 모델 크기, 보통 non-embedding parameter 수 |
| `D` | 데이터셋 크기, token 수 |
| `C` | 학습 compute |
| `L` | cross-entropy loss |

---

## 핵심 개념 2: Power Law

### 결론부터

**Power law는 x를 키우면 y가 일정한 지수 규칙에 따라 줄거나 커지는 관계입니다.**

아주 단순화하면 이런 형태입니다.

```text
Loss ≈ A × N^(-α) + irreducible loss
```

여기서:

| 기호 | 의미 |
|---|---|
| `N` | 모델 크기 |
| `α` | scaling exponent |
| `A` | 상수 |
| `irreducible loss` | 아무리 키워도 남는 손실에 가까운 항 |

쉽게 말하면:

```text
모델을 크게 만들면 loss는 줄어든다.
하지만 줄어드는 속도는 점점 느려진다.
```

예를 들어 모델을 10배 키웠다고 loss가 10배 좋아지는 것은 아닙니다. 보통은 조금씩, 하지만 매우 예측 가능하게 좋아집니다.

---

## 쉬운 비유: 공부 시간과 시험 점수

스케일링 법칙을 공부 시간으로 비유해봅시다.

처음 10시간 공부하면 점수가 크게 오릅니다.

```text
40점 → 70점
```

하지만 이미 많이 공부한 상태에서 10시간을 더 한다고 같은 폭으로 오르지는 않습니다.

```text
90점 → 92점
```

즉, 투입을 늘릴수록 성능은 좋아지지만, 개선폭은 점점 작아집니다.

LLM도 비슷합니다.

```text
모델 크기 증가
데이터 증가
컴퓨트 증가
→ loss 감소
```

하지만 그 감소는 점점 완만해집니다.

그래도 중요한 점은:

> 완만해지긴 하지만, 꽤 규칙적으로 좋아진다.

입니다.

---

## 핵심 개념 3: 모델 크기 N

### 결론부터

**모델 파라미터 수가 커질수록 loss가 예측 가능한 방식으로 줄어듭니다.**

작은 모델은 언어 패턴을 적게 저장하고 표현합니다.

```text
작은 모델:
문법, 단어 의미, 문장 구조 일부만 학습
```

큰 모델은 더 복잡한 패턴을 더 많이 담을 수 있습니다.

```text
큰 모델:
문법, 의미, 상식, 장거리 의존성, 다양한 스타일, task 패턴까지 더 잘 학습
```

---

## 핵심 개념 4: 데이터 크기 D

### 결론부터

**모델이 커져도 데이터가 부족하면 성능 향상이 막힙니다.**

큰 모델은 많은 것을 배울 수 있지만, 배울 데이터가 부족하면 같은 데이터를 반복해서 외우게 됩니다.

예를 들어 학생이 아주 똑똑해도 문제집이 10문제밖에 없으면 한계가 있습니다.

```text
능력은 큰데, 볼 문제가 부족함
→ 같은 문제를 외움
→ 새로운 문제에 약함
```

언어 모델도 마찬가지입니다.

```text
모델 크기 N 증가
데이터 D 고정
→ 어느 순간 overfitting / data-limited regime
```

---

## 핵심 개념 5: Compute C

### 결론부터

**컴퓨트는 모델을 학습하는 데 실제로 쓴 계산량입니다.**

모델이 크고 데이터가 많아도, 충분히 학습할 compute가 없으면 성능이 나오지 않습니다.

학습 compute는 대략 이런 요소에 영향을 받습니다.

```text
모델 크기
학습 token 수
batch size
training step 수
forward/backward 계산량
```

쉽게 말하면 compute는 이런 질문입니다.

> 모델을 얼마나 많이, 얼마나 오래, 얼마나 큰 규모로 학습했는가?

다만 compute는 단순 시간과는 다릅니다. 같은 연산량도 GPU를 많이 쓰면 wall-clock time은 줄어들 수 있습니다.

---

## 세 가지를 한 번에 이해하기

LLM 성능은 세 가지가 같이 맞아야 합니다.

```text
모델 크기 N
데이터 크기 D
컴퓨트 C
```

하나만 키워서는 안 됩니다.

| 상황 | 문제 |
|---|---|
| 모델은 큰데 데이터가 부족 | 데이터를 외우거나 data-limited |
| 데이터는 많은데 모델이 작음 | 모델 용량 부족 |
| 모델과 데이터는 큰데 compute 부족 | 충분히 학습 못 함 |
| compute는 큰데 모델/데이터 배분이 나쁨 | 비효율적 학습 |

이 논문의 핵심 메시지는:

> 성능을 높이려면 N, D, C를 함께 고려해야 한다.

입니다.

---

## 핵심 개념 6: 모델 shape보다 scale이 중요하다

이 논문에서 꽤 중요한 관찰이 나옵니다.

**같은 parameter 수라면, depth와 width의 세부 배분은 생각보다 덜 중요했습니다.**

예를 들어 두 모델이 있다고 합시다.

```text
모델 A: 깊고 좁음
모델 B: 얕고 넓음
```

둘의 parameter 수가 비슷하면, 성능 차이가 아주 크지 않았다는 것입니다.

이 말은 이렇게 이해하면 됩니다.

> 세부 architecture 튜닝도 중요하지만, 이 논문이 본 범위에서는 “얼마나 큰 모델인가”가 더 큰 요인이었다.

물론 이것이 “architecture는 전혀 중요하지 않다”는 뜻은 아닙니다. 같은 Transformer 계열 안에서, 합리적인 범위의 shape를 비교했을 때 scale의 영향이 훨씬 강했다는 뜻입니다.

---

## 핵심 개념 7: 큰 모델은 sample-efficient하다

### 결론부터

**큰 모델은 같은 성능에 도달하는 데 더 적은 data point나 optimization step을 필요로 할 수 있습니다.**

처음 들으면 이상합니다.

큰 모델은 더 많은 데이터가 필요할 것 같은데, 왜 sample-efficient하다고 할까요?

이 논문의 의미는 다음에 가깝습니다.

```text
같은 loss 수준에 도달하려면,
큰 모델이 작은 모델보다 더 적은 training step이나 data exposure로 도달할 수 있다.
```

예를 들어 작은 모델은 어떤 수준의 loss에 도달하려면 아주 오래 학습해야 할 수 있습니다.

```text
작은 모델:
많이 봐도 천천히 좋아짐
```

큰 모델은 더 빠르게 낮은 loss에 도달할 수 있습니다.

```text
큰 모델:
적은 step으로도 빠르게 좋은 표현을 학습
```

---

## 핵심 개념 8: 수렴까지 학습하는 것은 비효율적일 수 있다

이 논문의 유명한 주장 중 하나입니다.

**고정된 compute budget이 있을 때, 작은 모델을 끝까지 수렴시킬 바에는 큰 모델을 적당히 학습하고 일찍 멈추는 편이 더 효율적일 수 있다.**

예를 들어 compute 예산이 정해져 있습니다.

```text
총 GPU 시간 = 100
```

선택지는 두 가지입니다.

### 선택 A

```text
작은 모델
오래 학습
거의 수렴까지 학습
```

### 선택 B

```text
큰 모델
짧게 학습
수렴 전 early stopping
```

Kaplan scaling law 관점에서는, compute-efficient하게는 선택 B가 더 나을 수 있다고 봅니다.

---

## 단, 여기에는 중요한 후속 수정이 있다

현재 관점에서는 이 논문을 읽을 때 반드시 알아야 할 점이 있습니다.

**Kaplan et al.의 scaling law는 이후 Chinchilla 논문에서 compute-optimal allocation 관점이 수정되었습니다.**

Kaplan scaling law는 compute-efficient training에서 모델 크기에 더 많은 비중을 두고, 상대적으로 적은 데이터로 큰 모델을 일찍 멈추는 쪽을 강조했습니다. 반면 2022년 DeepMind의 Chinchilla 논문은 같은 compute budget에서 모델 크기와 학습 token 수를 더 균형 있게 키워야 하며, 모델 크기를 두 배로 키우면 학습 token 수도 두 배로 키우는 것이 compute-optimal하다고 주장했습니다.

즉, 오늘 보는 Kaplan 논문은 역사적으로 매우 중요하지만, 현재 실무적 결론은 이렇게 업데이트해서 이해하는 것이 좋습니다.

```text
Kaplan et al.:
큰 모델을 쓰고, 수렴 전 멈추는 것이 compute-efficient할 수 있다.

Chinchilla:
큰 모델만 키우지 말고, 모델 크기와 학습 token 수를 균형 있게 키워야 한다.
```

## 쉬운 예시: 공부 잘하는 학생과 문제집

## 모델 크기 N

학생의 두뇌 용량입니다.

```text
작은 모델 = 기억력과 이해력이 제한된 학생
큰 모델 = 더 많은 패턴을 이해할 수 있는 학생
```

## 데이터 크기 D

문제집의 양입니다.

```text
적은 데이터 = 문제 수가 적음
많은 데이터 = 다양한 문제를 많이 봄
```

## compute C

공부 시간과 연산 자원입니다.

```text
compute = 학생이 문제를 풀고 피드백 받는 총 시간
```

성적을 잘 내려면 세 가지가 모두 필요합니다.

```text
똑똑한 학생
충분한 문제집
충분한 공부 시간
```

## 이 논문을 오해하면 안 되는 부분

### 오해 1. “모델만 키우면 된다”

아닙니다.

모델 크기, 데이터 크기, compute를 함께 봐야 합니다.

### 오해 2. “성능이 무한히 좋아진다”

아닙니다.

관찰된 범위에서 loss가 계속 줄어드는 경향을 보였다는 뜻입니다.

### 오해 3. “Architecture는 안 중요하다”

그렇게까지 말하면 안 됩니다.

이 논문은 주로 Transformer 계열 안에서 shape 변화를 봤고, 합리적인 범위에서 영향이 작았다고 본 것입니다.

### 오해 4. “Kaplan scaling law가 지금도 최종 결론이다”

아닙니다.

Kaplan scaling law는 매우 중요한 출발점이지만, compute-optimal training에 대한 결론은 이후 Chinchilla에서 수정됐습니다.

---

## 실무적으로 이 논문에서 배울 점

## 1. 작은 실험을 대충 보면 안 된다

작은 모델을 여러 크기와 데이터 조건에서 체계적으로 학습하면, 큰 모델을 학습하기 전 중요한 예측을 할 수 있습니다.

```text
작은 scale 실험
→ scaling curve fitting
→ 큰 scale 성능 예측
```

## 2. 모델 크기, 데이터, compute는 따로 보면 안 된다

예를 들어 모델 성능이 나쁘다고 바로 모델만 키우면 안 됩니다.

먼저 물어야 합니다.

```text
모델이 작은가?
데이터가 부족한가?
compute가 부족한가?
학습을 너무 짧게 했나?
데이터 재사용이 과한가?
```

## 3. Loss는 downstream 성능의 완전한 대체 지표는 아니지만 중요하다

language modeling loss는 여전히 모델 성능의 중요한 기본 지표입니다.

---

## 이 논문이 주는 진짜 메시지

이 논문의 진짜 메시지는 단순히 “큰 모델이 좋다”가 아닙니다.

더 깊은 메시지는 이것입니다.

**LLM 성능 향상은 어느 정도 예측 가능한 공학적 scaling 문제로 다룰 수 있다.**

이전에는 모델을 크게 만들면 성능이 좋아질지, 어디서 막힐지, compute를 어떻게 써야 할지 불확실했습니다.

이 논문은 다음과 같은 관점을 제시했습니다.

```text
작은 scale에서 법칙을 찾고,
그 법칙으로 큰 scale을 예측하고,
모델 크기·데이터·compute를 배분한다.
```

## 입문자가 꼭 기억해야 할 문장

**Scaling Laws는 언어 모델의 loss가 모델 크기, 데이터 크기, 학습 compute를 키울수록 power-law 형태로 예측 가능하게 감소한다는 것을 보여준 논문입니다.**

더 짧게 말하면:

> **Scaling Laws = LLM은 크게 만들수록 좋아지는데, 그 좋아지는 정도가 꽤 예측 가능하다는 법칙**

## 오늘 공부용 요약

**논문명:** Scaling Laws for Neural Language Models  
**저자:** Jared Kaplan, Sam McCandlish, Tom Henighan, Tom B. Brown, Benjamin Chess, Rewon Child, Scott Gray, Alec Radford, Jeffrey Wu, Dario Amodei  
**핵심 결론:** Transformer 언어 모델의 cross-entropy loss는 모델 크기 `N`, 데이터 크기 `D`, 학습 compute `C`에 대해 power-law 형태로 감소한다.

**왜 중요함:** GPT-3 이후 LLM 개발에서 “모델을 얼마나 키우면 얼마나 좋아질까?”라는 질문을 정량적으로 다루게 만든 핵심 논문입니다.

**입문자가 배울 점:** 모델 크기만 보면 안 됩니다. 좋은 LLM을 만들려면 모델 크기, 데이터 token 수, compute budget을 함께 봐야 합니다.

**가장 중요한 주의점:** Kaplan scaling law의 compute-optimal 결론은 이후 Chinchilla 논문에서 수정되었습니다. 따라서 현재 관점에서는 “scale이 중요하다”는 큰 메시지는 유지하되, “모델과 데이터의 균형”은 Chinchilla까지 함께 읽어야 합니다.
