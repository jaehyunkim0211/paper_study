# Datasheets for Datasets — Gebru et al.

## 결론부터

이 논문의 결론은 이겁니다.

**머신러닝 데이터셋도 제품 설명서처럼 “데이터 설명서”를 가져야 한다.**  
데이터셋이 어떻게 만들어졌는지, 무엇을 포함하는지, 어떤 편향이나 한계가 있는지, 어떤 용도로 써도 되고 어떤 용도로 쓰면 안 되는지를 문서화해야 한다는 주장입니다.

논문에서 말하는 **datasheet**는 쉽게 말해 이런 것입니다.

> “이 데이터셋은 누가, 왜, 어떻게 만들었고, 무엇이 들어 있으며, 어떤 전처리를 했고, 어떤 용도로 쓰는 것이 적절한가?”

저자들은 머신러닝 커뮤니티에 데이터셋 문서화의 표준 절차가 부족하다고 보고, 전자부품에 datasheet가 붙는 것처럼 데이터셋에도 motivation, composition, collection process, recommended uses 등을 기록한 문서가 붙어야 한다고 제안합니다.

---

## 이 논문을 한 문장으로 요약하면

**모델의 문제는 종종 모델 자체가 아니라, 제대로 이해하지 못한 데이터에서 시작된다.**

이 논문은 “좋은 모델을 만들자”보다 한 단계 앞선 질문을 던집니다.

> 우리가 쓰는 데이터는 믿을 만한가?  
> 이 데이터가 어떤 맥락에서 만들어졌는지 알고 있는가?  
> 이 데이터로 학습한 모델을 어디에 써도 되는지 알고 있는가?

데이터는 머신러닝 모델의 학습과 평가에 핵심적이고, 데이터셋의 특성은 모델의 행동에 직접적인 영향을 줍니다. 학습·평가 데이터와 실제 배포 환경이 맞지 않거나 데이터에 사회적 편향이 반영되어 있으면, 고위험 영역에서 심각한 결과가 생길 수 있습니다.

---

## 왜 이 논문이 중요한가?

이전 논문들을 떠올려보면 흐름이 보입니다.

| 논문 | 핵심 질문 |
|---|---|
| **Tidy Data** | 데이터를 분석하기 좋은 구조로 만들었는가? |
| **Two Cultures** | 설명이 목적인가, 예측이 목적인가? |
| **Model Evaluation** | 모델을 공정하게 평가했는가? |
| **Datasheets for Datasets** | 애초에 이 데이터셋을 제대로 이해하고 있는가? |

이 논문은 데이터사이언스에서 매우 중요한 관점을 줍니다.

**데이터셋은 그냥 “주어진 것”이 아니다.**  
데이터셋도 누군가의 선택, 수집 방식, 제외 기준, 라벨링 기준, 전처리 방식, 배포 방식의 결과물입니다.

예를 들어 어떤 얼굴 이미지 데이터셋이 있다고 해봅시다.

그 데이터셋에 대해 다음 정보를 모른다면 위험합니다.

| 질문 | 왜 중요한가 |
|---|---|
| 어떤 국가, 인종, 성별, 연령대의 이미지가 많은가? | 특정 집단에서 성능이 낮을 수 있음 |
| 이미지는 어떤 상황에서 수집됐는가? | 실제 배포 환경과 다를 수 있음 |
| 동의는 받았는가? | 개인정보·윤리 문제가 생길 수 있음 |
| 라벨은 누가 달았는가? | 라벨 오류나 편견이 들어갈 수 있음 |
| 어떤 용도로 쓰면 안 되는가? | 위험한 오용을 막아야 함 |

즉, datasheet는 단순한 문서가 아니라 **데이터셋 사용 전 위험 점검표**입니다.

---

## 핵심 개념 1: 데이터셋은 모델의 행동을 결정한다

**모델은 데이터에서 배운다. 그러므로 데이터셋의 결함은 모델의 결함으로 이어진다.**

예를 들어 고객 이탈 예측 모델을 만든다고 해봅시다.

데이터셋이 다음과 같다면 문제가 생깁니다.

| 데이터 문제 | 모델에 생기는 문제 |
|---|---|
| 특정 연령대 고객이 거의 없음 | 해당 연령대 예측이 부정확함 |
| 이탈 사유가 기록되지 않음 | 모델 해석이 빈약해짐 |
| 과거 VIP 고객만 포함됨 | 일반 고객에게 적용하기 어려움 |
| 라벨 기준이 자주 바뀜 | 학습 타깃이 일관되지 않음 |
| 결측치를 임의로 0으로 채움 | 잘못된 패턴을 학습할 수 있음 |

모델을 아무리 잘 튜닝해도, 데이터셋 자체가 잘못되어 있으면 한계가 있습니다.

이 논문이 말하는 중요한 메시지는:

> 데이터 품질은 단순히 accuracy를 높이기 위한 문제가 아니라, 책임성과 안전성의 문제다.

입니다.

---

## 핵심 개념 2: datasheet는 데이터셋의 “사용 설명서”다

논문은 데이터셋 문서화를 위해 질문들을 제안합니다. 이 질문들은 데이터셋 생애주기와 맞춰 구성됩니다. 크게 보면 **motivation, composition, collection process, preprocessing/cleaning/labeling, uses, distribution, maintenance** 항목으로 나뉩니다.

쉽게 말하면 다음 7가지를 묻는 것입니다.

| 항목 | 쉽게 말하면 | 대표 질문 |
|---|---|---|
| **Motivation** | 왜 만들었는가? | 어떤 목적·태스크를 위해 만들었나? |
| **Composition** | 무엇이 들어 있는가? | 샘플 수, 라벨, 결측치, 오류, 데이터 split은? |
| **Collection Process** | 어떻게 수집했는가? | 누가, 어디서, 어떤 방식으로 수집했나? |
| **Preprocessing / Cleaning / Labeling** | 어떻게 가공했는가? | 전처리, 라벨링, 제거 기준은 무엇인가? |
| **Uses** | 어디에 써도 되는가? | 권장 용도와 비권장 용도는 무엇인가? |
| **Distribution** | 어떻게 배포되는가? | 접근 권한, 라이선스, 제약 조건은? |
| **Maintenance** | 누가 관리하는가? | 업데이트, 오류 수정, 연락처는? |

여기서 핵심은 “데이터셋을 많이 설명하자”가 아니라, **데이터셋 사용자가 올바른 의사결정을 할 수 있을 만큼 충분히 설명하자**입니다.

---

## 핵심 개념 3: 데이터셋 생성자와 소비자를 연결한다

이 논문은 데이터셋과 관련된 사람을 크게 두 그룹으로 봅니다.

| 역할 | 의미 |
|---|---|
| **Dataset creator** | 데이터셋을 만든 사람 또는 조직 |
| **Dataset consumer** | 데이터셋을 가져다 쓰는 사람 또는 조직 |

문제는 creator가 알고 있는 정보가 consumer에게 전달되지 않는다는 것입니다.

예를 들어 creator는 이런 사실을 알고 있을 수 있습니다.

> “이 데이터셋은 원래 연구용으로만 만든 것이다.”  
> “라벨링 품질이 특정 카테고리에서 낮다.”  
> “특정 집단 샘플이 부족하다.”  
> “상업적 사용은 라이선스상 불가능하다.”  
> “테스트셋은 특정 기간 데이터라서 장기 일반화 성능을 보장하지 않는다.”

그런데 consumer가 이런 정보를 모르면 잘못 사용할 수 있습니다.

Datasheet는 데이터셋 생성자에게는 생성·배포·유지 과정, 가정, 잠재적 위험을 성찰하게 하고, 데이터셋 소비자에게는 특정 작업에 그 데이터셋을 써도 되는지 판단할 정보를 제공하는 역할을 합니다.

---

## 쉬운 예시: 채용 모델용 이력서 데이터셋

채용 추천 모델을 만든다고 해봅시다. 어떤 이력서 데이터셋을 받았습니다.

그냥 보면 이런 정보만 있을 수 있습니다.

| 지원자 ID | 경력연수 | 학교 | 기술스택 | 합격 여부 |
|---|---:|---|---|---|
| A001 | 3 | X대학 | Python, SQL | 합격 |
| A002 | 1 | Y대학 | Excel, R | 불합격 |

겉으로는 모델 학습이 가능합니다.

하지만 datasheet 관점에서는 다음 질문을 해야 합니다.

| 질문 | 이유 |
|---|---|
| 이 데이터는 어느 회사, 어느 직무의 채용 데이터인가? | 다른 회사·직무에 일반화하기 어려울 수 있음 |
| 합격 여부는 실제 성과를 의미하는가, 면접관 판단을 의미하는가? | 라벨이 편향을 포함할 수 있음 |
| 특정 성별, 연령, 학교, 지역이 과소대표되었는가? | 차별적 모델이 될 수 있음 |
| 과거 채용 방식에 편향이 있었는가? | 모델이 과거 편향을 재생산할 수 있음 |
| 개인정보 동의와 보관 정책은 무엇인가? | 법적·윤리적 문제가 있음 |
| 이 데이터로 자동 채용 판단을 해도 되는가? | 사용 목적 제한이 필요할 수 있음 |

이 정보가 없으면 모델 성능이 높아도 위험합니다.

예를 들어 과거 채용 데이터가 특정 학교 출신을 선호했다면, 모델은 그 패턴을 “좋은 지원자의 특징”으로 학습할 수 있습니다. 이 경우 accuracy가 높아도 공정한 모델이라고 말하기 어렵습니다.

## 핵심 개념 4: “어떤 용도로 쓰면 안 되는가”도 문서화해야 한다

Datasheet에서 특히 중요한 부분은 **Uses**입니다.

데이터셋 설명서는 단순히 “이 데이터셋은 이런 용도로 좋습니다”만 말하면 안 됩니다.

반대로:

> “이 데이터셋을 이런 용도로 쓰면 안 됩니다.”

도 말해야 합니다.

예를 들어 어떤 감정 분석 데이터셋이 영화 리뷰로 만들어졌다고 합시다.

| 사용 목적 | 적절성 |
|---|---|
| 영화 리뷰 긍정/부정 분류 | 적절 |
| 제품 리뷰 감정 분석 | 어느 정도 가능하지만 도메인 차이 확인 필요 |
| 의료 상담 감정 판단 | 부적절할 가능성 큼 |
| 직원 성과 평가 | 매우 부적절 |

왜냐하면 영화 리뷰에서의 “긍정/부정”과 의료 상담, 인사 평가, 정신건강 맥락에서의 감정은 전혀 다를 수 있기 때문입니다.

---

## 핵심 개념 5: datasheet는 자동 생성 문서가 아니다

요즘은 자동 문서화 도구가 많습니다. 데이터 컬럼 수, 결측치 비율, 분포, 상관관계 등을 자동으로 뽑을 수 있습니다.

그런데 이 논문이 말하는 datasheet는 단순한 자동 리포트가 아닙니다.

자동으로 뽑을 수 있는 정보:

| 자동 리포트 가능 |
|---|
| 행 수, 열 수 |
| 결측치 비율 |
| 변수별 분포 |
| 라벨 비율 |
| 중복 행 수 |
| 기본 통계량 |

하지만 자동으로 알기 어려운 정보:

| 사람이 설명해야 하는 정보 |
|---|
| 왜 이 데이터셋을 만들었는가 |
| 어떤 데이터가 제외되었는가 |
| 수집 과정에서 어떤 선택이 있었는가 |
| 라벨링 기준은 어떻게 정했는가 |
| 어떤 집단이 과소대표되었는가 |
| 어떤 용도로 쓰면 위험한가 |
| 누가 유지보수 책임을 지는가 |

즉, datasheet는 **통계 요약표 + 맥락 설명 + 책임 기록**이 합쳐진 문서입니다.

---

## 실무 예시: 고객 이탈 데이터셋 datasheet를 만든다면

고객 이탈 예측 데이터를 만든다고 해봅시다.

datasheet에는 이런 내용이 들어가야 합니다.

### 1. Motivation

| 질문 | 예시 답변 |
|---|---|
| 데이터셋을 만든 목적은? | 구독 서비스 고객 이탈 예측 |
| 특정 태스크가 있었나? | 다음 30일 내 이탈 여부 예측 |
| 누가 만들었나? | 데이터팀, CRM팀 공동 |
| 누가 비용을 지원했나? | 마케팅 조직 |

### 2. Composition

| 질문 | 예시 답변 |
|---|---|
| 관측치는 무엇인가? | 고객 1명 기준 월별 스냅샷 |
| 샘플 수는? | 고객-월 단위 120만 건 |
| 라벨은 무엇인가? | 해당 월 이후 30일 내 구독 해지 여부 |
| 결측치는 있는가? | 일부 고객의 상담 기록 누락 |
| 추천 split은? | 시간 기준 train/validation/test split |

### 3. Collection Process

| 질문 | 예시 답변 |
|---|---|
| 어디서 수집했는가? | 결제 DB, 사용 로그, 상담 로그 |
| 수집 기간은? | 2022년 1월부터 2025년 12월 |
| 제외된 고객은? | 법인 고객, 무료 체험 고객 |

### 4. Preprocessing / Labeling

| 질문 | 예시 답변 |
|---|---|
| 전처리는 무엇을 했는가? | 이상 결제금액 제거, 로그 집계 |
| 라벨은 어떻게 만들었는가? | 해지 이벤트 발생 여부 기준 |
| 원본 데이터는 보관했는가? | 별도 저장소에 보관 |

### 5. Uses

| 질문 | 예시 답변 |
|---|---|
| 권장 용도는? | 이탈 위험 고객 우선순위화 |
| 비권장 용도는? | 고객 신용평가, 가격 차별 |
| 주의할 점은? | 신규 요금제 출시 이후 분포 변화 가능 |

### 6. Distribution / Maintenance

| 질문 | 예시 답변 |
|---|---|
| 누가 접근 가능한가? | 승인된 데이터사이언스·CRM 담당자 |
| 업데이트 주기는? | 매월 |
| 담당자는? | 데이터 플랫폼팀 |

이 정도 문서가 있으면 모델러는 훨씬 안전하게 판단할 수 있습니다.

## 이 논문이 말하는 “좋은 데이터셋”의 기준

좋은 데이터셋은 단순히 큰 데이터셋이 아닙니다.

| 좋은 데이터셋의 특징 | 설명 |
|---|---|
| 목적이 명확하다 | 왜 만들어졌는지 설명되어 있음 |
| 구성 정보가 투명하다 | 무엇이 들어 있고 무엇이 빠졌는지 알 수 있음 |
| 수집 과정이 기록되어 있다 | 데이터 생성 맥락을 이해할 수 있음 |
| 전처리와 라벨링이 설명되어 있다 | 어떤 변환과 판단이 있었는지 알 수 있음 |
| 적절한 사용 범위가 명시되어 있다 | 오용을 줄일 수 있음 |
| 유지보수 계획이 있다 | 오래된 데이터나 오류를 관리할 수 있음 |

즉, 데이터셋의 품질은 단순히 크기나 모델 성능으로만 판단하면 안 됩니다.

**문서화 가능성, 추적 가능성, 책임 가능성**도 품질의 일부입니다.

---

## 이 논문의 한계도 알아야 한다

이 논문의 주장은 매우 중요하지만, 실제 적용에는 어려움이 있습니다.

| 어려움 | 설명 |
|---|---|
| 시간이 많이 든다 | datasheet 작성은 단순 자동화 리포트보다 비용이 큼 |
| 모든 질문에 답하기 어렵다 | 오래된 데이터셋은 수집 경위를 모를 수 있음 |
| 조직 내 책임자가 불분명할 수 있다 | 데이터 소유권과 유지보수 주체가 애매함 |
| 법적·윤리적 판단이 어렵다 | 기술팀만으로 결정하기 어려운 문제가 있음 |
| 문서가 있어도 사용자가 안 읽을 수 있다 | 프로세스에 통합되어야 효과가 있음 |

그래서 이 논문의 현실적인 해석은 이겁니다.

> 완벽한 datasheet를 만들 수 없더라도, 아무 문서 없이 데이터셋을 쓰는 것보다는 훨씬 낫다.

---

## 현대 데이터사이언스와 LLM 시대에서의 의미

이 논문은 2018년에 처음 나왔지만, 지금은 더 중요해졌습니다. 특히 LLM과 파운데이션 모델 시대에는 데이터셋의 규모가 커지고, 수집 경로가 복잡해지고, 저작권·개인정보·편향·안전성 문제가 더 커졌습니다.

예를 들어 LLM 학습 데이터에 대해 이런 질문이 필요합니다.

| 질문 | LLM에서 중요한 이유 |
|---|---|
| 어떤 웹사이트와 문서가 포함됐는가? | 지식 범위와 편향에 영향 |
| 어떤 언어가 얼마나 포함됐는가? | 언어별 성능 차이에 영향 |
| 개인정보는 제거됐는가? | 프라이버시 위험 |
| 저작권 자료는 어떻게 처리됐는가? | 법적 위험 |
| 유해 콘텐츠는 어떻게 필터링했는가? | 안전성 문제 |
| 평가 데이터와 학습 데이터가 겹치지 않는가? | 벤치마크 오염 위험 |

즉, datasheet의 핵심 정신은 지금도 그대로 유효합니다.

**모델이 커질수록 데이터 문서화는 더 중요해집니다.**

---

## 이전 논문들과 연결해서 이해하기

### Tidy Data와의 연결

Tidy Data는 데이터의 **형태**를 봅니다.

> 변수는 열, 관측치는 행, 관측 단위는 테이블.

Datasheets for Datasets는 데이터의 **맥락**을 봅니다.

> 이 데이터는 왜 만들어졌고, 어떻게 수집됐고, 어디에 써야 하는가?

| 관점 | 질문 |
|---|---|
| Tidy Data | 데이터 구조가 분석하기 좋은가? |
| Datasheets | 데이터 맥락이 책임 있게 문서화되었는가? |

### Model Evaluation 논문과의 연결

Model Evaluation 논문은 test set과 validation set을 엄격히 구분하라고 했습니다.

Datasheets for Datasets는 그보다 앞서 이렇게 묻습니다.

> 이 train/validation/test split은 어떻게 만들어졌는가?  
> 이 test set은 실제 배포 환경을 대표하는가?  
> 어떤 데이터가 제외되었는가?

즉, 평가 설계도 데이터셋 문서화 없이는 불완전합니다.

## 입문자가 이 논문에서 꼭 가져가야 할 실전 규칙

### 1. 데이터셋을 받으면 먼저 출처부터 확인하라

모델부터 돌리지 말고, 먼저 물어야 합니다.

> 이 데이터는 어디서 왔는가?

출처가 불분명한 데이터는 성능이 좋아도 위험합니다.

### 2. 라벨 정의를 반드시 확인하라

타깃이 `churn = 1`이라고 되어 있어도, 그 정의가 다를 수 있습니다.

| 가능한 정의 |
|---|
| 구독 해지 버튼을 누른 고객 |
| 결제를 실패한 고객 |
| 30일간 접속하지 않은 고객 |
| 고객센터에 해지 의사를 밝힌 고객 |

라벨 정의가 다르면 모델이 배우는 것도 달라집니다.

### 3. 데이터가 대표하는 범위를 확인하라

데이터셋이 전체 고객을 대표하지 않을 수 있습니다.

예를 들어 마케팅 캠페인 대상자만 포함된 데이터로 전체 고객 이탈 모델을 만들면, 배포 후 성능이 무너질 수 있습니다.

### 4. 전처리 기록을 확인하라

전처리 과정에서 중요한 정보가 사라질 수 있습니다.

예를 들어 이상치를 모두 제거했는데, 실제로는 그 이상치가 사기 거래였다면 사기 탐지 모델에는 치명적입니다.

### 5. “이 데이터로 하면 안 되는 일”을 명확히 하라

권장 용도보다 더 중요한 것이 비권장 용도일 때가 있습니다.

예를 들어 고객 행동 데이터는 추천 모델에는 쓸 수 있지만, 민감한 신용평가나 고용 판단에 쓰면 부적절할 수 있습니다.

## 오늘 공부용 요약

**논문명:** Datasheets for Datasets  
**저자:** Timnit Gebru, Jamie Morgenstern, Briana Vecchione, Jennifer Wortman Vaughan, Hanna Wallach, Hal Daumé III, Kate Crawford  
**핵심 결론:** 데이터셋도 제품 설명서처럼 문서화되어야 한다. 데이터셋의 목적, 구성, 수집 과정, 전처리, 라벨링, 권장 사용처, 비권장 사용처, 배포 방식, 유지보수 계획을 기록해야 한다.  
**왜 중요함:** 모델은 데이터에서 배우기 때문에, 데이터셋의 편향·누락·수집 맥락·오용 가능성을 모르면 모델 성능과 책임성을 제대로 판단할 수 없다.  
**입문자가 배울 점:** 모델링 전에 데이터셋의 출처, 대표성, 라벨 정의, 수집 방식, 사용 제한을 먼저 확인해야 한다.  
**가장 중요한 문장:** 좋은 데이터사이언티스트는 모델만 평가하지 않고, 모델이 배운 데이터셋도 평가한다.
