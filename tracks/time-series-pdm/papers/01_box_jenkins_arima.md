# 1편: The Box-Jenkins Approach to Time Series Analysis and Forecasting: Principles and Applications

## 결론부터

**이 논문의 핵심 결론은, 시계열 예측은 감으로 하는 것이 아니라 “정상성 확인 → 차분/변환 → ARIMA 모델 후보 설정 → 잔차 진단 → 예측”이라는 반복 가능한 절차로 다뤄야 한다는 것입니다.**

## 이 논문을 한 문장으로 요약하면

**Box-Jenkins 접근법은 과거 값과 과거 오차를 이용해 미래를 예측하는 ARIMA 모델을 체계적으로 만들고 검증하는 방법론입니다.** 이 논문은 Box-Jenkins 표기법, 모델 식별, 추정, 진단, 예측, 계절형 모델까지의 절차를 소개하는 입문용 성격이 강합니다.

참고 링크: https://www.numdam.org/item/RO_1977__11_2_129_0.pdf

---

## 왜 이 논문이 필요했나?

시계열 데이터는 일반적인 표 데이터와 다릅니다.

일반적인 표 데이터에서는 이런 식으로 생각합니다.

| 온도 | 압력 | 전류 | 고장 여부 |
|---:|---:|---:|---|
| 72.1 | 3.1 | 10.2 | 정상 |
| 85.4 | 3.8 | 13.9 | 고장 |

각 행이 비교적 독립적인 샘플처럼 보입니다.

그런데 시계열은 다릅니다.

| 시간 | 온도 |
|---|---:|
| 09:00 | 70.0 |
| 09:01 | 70.2 |
| 09:02 | 70.4 |
| 09:03 | 70.9 |
| 09:04 | 71.1 |
| 09:05 | 71.3 |

여기서 09:05의 온도는 09:04, 09:03, 09:02와 강하게 연결되어 있습니다. 즉, **시간 순서 자체가 정보**입니다.

Box-Jenkins 이전에도 예측은 가능했습니다. 하지만 문제는 “어떤 모델을 고를지”, “모델이 잘 맞는지 어떻게 확인할지”, “예측 오차를 어떻게 해석할지”가 체계적이지 않았다는 점입니다. 이 논문이 중요한 이유는 시계열 모델링을 하나의 작업 절차로 정리했기 때문입니다.

설비 데이터 관점에서 보면 더 중요합니다. 온도, 압력, 전류, 진동 RMS 같은 센서값은 대부분 시간 의존성이 있습니다. 정상 설비도 어제의 상태, 1분 전의 상태, 10초 전의 상태에 영향을 받습니다. 그래서 “현재 값이 높다/낮다”만 보는 것보다, **평소 패턴에서 얼마나 벗어났는지**를 봐야 합니다. ARIMA는 그 출발점입니다.

---

## 핵심 개념 1. 시계열은 “순서가 있는 데이터”다

가장 먼저 가져야 할 감각은 이것입니다.

**시계열에서 행을 섞으면 정보가 망가집니다.**

예를 들어 아래 온도 데이터가 있다고 하겠습니다.

| 시간 | 온도 |
|---|---:|
| 09:00 | 70.0 |
| 09:01 | 70.2 |
| 09:02 | 70.4 |
| 09:03 | 70.9 |
| 09:04 | 71.1 |
| 09:05 | 71.3 |

이 데이터를 랜덤하게 섞으면 다음처럼 됩니다.

| 시간 | 온도 |
|---|---:|
| 09:04 | 71.1 |
| 09:01 | 70.2 |
| 09:05 | 71.3 |
| 09:00 | 70.0 |
| 09:03 | 70.9 |
| 09:02 | 70.4 |

표 데이터 모델은 이걸 크게 신경 쓰지 않을 수 있지만, 시계열 모델에서는 치명적입니다. “점점 올라가는 추세”라는 정보가 사라졌기 때문입니다.

설비 진단에서도 마찬가지입니다. 베어링 진동 데이터에서 고장 직전에는 특정 주파수 성분이 커지거나 진동 RMS가 서서히 증가할 수 있습니다. 이때 중요한 것은 단일 시점 값이 아니라 **시간에 따른 변화**입니다.

---

## 핵심 개념 2. 정상성: 모델링하기 쉬운 상태로 만드는 것

ARIMA의 중요한 전제 중 하나는 **정상성(stationarity)**입니다.

쉽게 말하면 정상성은 이런 뜻입니다.

**시계열의 평균, 분산, 자기상관 구조가 시간이 지나도 크게 변하지 않는 상태**입니다.

예를 들어 아래 데이터는 정상적이지 않습니다.

| 시간 | 온도 |
|---|---:|
| 09:00 | 70.0 |
| 09:01 | 70.5 |
| 09:02 | 71.0 |
| 09:03 | 71.5 |
| 09:04 | 72.0 |
| 09:05 | 72.5 |

계속 올라가고 있습니다. 평균이 시간에 따라 변합니다.

반면 아래 데이터는 비교적 정상적입니다.

| 시간 | 온도 |
|---|---:|
| 09:00 | 70.1 |
| 09:01 | 69.9 |
| 09:02 | 70.2 |
| 09:03 | 70.0 |
| 09:04 | 69.8 |
| 09:05 | 70.1 |

70도 근처에서 흔들릴 뿐입니다.

ARIMA는 보통 원본 데이터가 비정상적이면 **차분(differencing)**해서 정상성에 가깝게 만듭니다.

---

## 핵심 개념 3. 차분: 값이 아니라 “변화량”을 보는 것

차분은 어렵게 들리지만 매우 단순합니다.

현재 값에서 바로 이전 값을 뺍니다.

| 시간 | 온도 | 1차 차분 |
|---|---:|---:|
| 09:00 | 70.0 | - |
| 09:01 | 70.5 | 0.5 |
| 09:02 | 71.0 | 0.5 |
| 09:03 | 71.4 | 0.4 |
| 09:04 | 71.8 | 0.4 |
| 09:05 | 72.1 | 0.3 |

원래 온도는 계속 증가하지만, 차분값은 0.3~0.5 근처에서 움직입니다. 즉, “온도 수준”이 아니라 “온도가 얼마나 빨리 변하는지”를 보는 것입니다.

설비 데이터에서도 유용합니다.

예를 들어 모터 온도가 60도에서 80도로 올라갔다고 해서 무조건 이상은 아닐 수 있습니다. 부하가 증가하면 온도도 올라갈 수 있기 때문입니다. 하지만 평소에는 1분에 0.1도씩 오르던 설비가 갑자기 1분에 2도씩 오르면 이상 신호일 수 있습니다.

---

## 핵심 개념 4. AR: 과거 값으로 현재를 설명한다

AR은 **AutoRegressive**, 즉 자기회귀입니다.

직관은 간단합니다.

**현재 값은 과거 값의 영향을 받는다.**

예를 들어 09:05의 온도는 09:04의 온도와 관련이 큽니다.

| 시간 | 온도 |
|---|---:|
| 09:03 | 70.9 |
| 09:04 | 71.1 |
| 09:05 | 71.3 |

온도 센서는 관성이 있기 때문에 갑자기 50도나 100도로 튀지 않습니다. 직전 값과 비슷할 가능성이 큽니다.

가장 단순한 AR(1)은 이렇게 씁니다.

\[
y_t = c + \phi y_{t-1} + \epsilon_t
\]

뜻은 다음과 같습니다.

| 기호 | 의미 |
|---|---|
| \(y_t\) | 현재 시점의 값 |
| \(y_{t-1}\) | 바로 이전 시점의 값 |
| \(c\) | 기본 수준 |
| \(\phi\) | 이전 값이 현재에 얼마나 영향을 주는지 |
| \(\epsilon_t\) | 설명하지 못한 오차 |

예를 들어 \(\phi\)가 크면 직전 값의 영향이 강합니다. 설비 온도, 압력, 전류 RMS처럼 천천히 움직이는 센서에서는 AR 성분이 강할 수 있습니다.

---

## 핵심 개념 5. MA: 과거 오차로 현재를 설명한다

MA는 **Moving Average**입니다.

여기서 “이동평균”이라고 해서 단순 rolling average만 생각하면 안 됩니다. ARIMA에서 MA는 **과거의 예측 오차가 현재 값에 영향을 준다**는 뜻입니다.

예를 들어 어느 순간 센서값이 예상보다 높게 나왔습니다.

| 시간 | 실제 온도 | 모델 예측 | 오차 |
|---|---:|---:|---:|
| 09:03 | 70.9 | 70.5 | +0.4 |
| 09:04 | 71.1 | 70.8 | +0.3 |
| 09:05 | 71.3 | 71.0 | +0.3 |

계속 모델이 낮게 예측하고 있습니다. 그러면 “최근 오차가 계속 양수였으니 다음 예측도 조금 올려야겠다”고 생각할 수 있습니다. 이게 MA의 직관입니다.

가장 단순한 MA(1)은 이렇게 씁니다.

\[
y_t = c + \epsilon_t + \theta \epsilon_{t-1}
\]

뜻은 다음과 같습니다.

| 기호 | 의미 |
|---|---|
| \(\epsilon_t\) | 현재 오차 |
| \(\epsilon_{t-1}\) | 바로 이전 오차 |
| \(\theta\) | 이전 오차의 영향력 |

---

## 핵심 개념 6. ARIMA: AR + 차분 + MA

ARIMA는 세 가지를 합친 이름입니다.

\[
ARIMA(p, d, q)
\]

| 요소 | 의미 | 쉬운 설명 |
|---|---|---|
| \(p\) | AR 차수 | 과거 값을 몇 시점까지 볼 것인가 |
| \(d\) | differencing 차수 | 몇 번 차분할 것인가 |
| \(q\) | MA 차수 | 과거 오차를 몇 시점까지 볼 것인가 |

예를 들어 ARIMA(1,1,1)은 이런 뜻입니다.

1. 원본 데이터를 한 번 차분한다.
2. 차분된 값에 대해 과거 1시점 값을 본다.
3. 과거 1시점의 오차도 반영한다.

조금 더 일반적인 식은 다음처럼 씁니다.

\[
\Phi(B)(1-B)^d y_t = c + \Theta(B)\epsilon_t
\]

처음 보면 복잡하지만, 직관은 단순합니다.

- \((1-B)^d\): 데이터를 차분해서 추세를 제거한다.
- \(\Phi(B)\): 과거 값의 영향을 반영한다.
- \(\Theta(B)\): 과거 오차의 영향을 반영한다.
- \(\epsilon_t\): 그래도 설명하지 못한 랜덤한 부분이다.

즉, ARIMA는 “추세를 제거한 뒤, 과거 값과 과거 오차로 현재를 설명하는 모델”입니다.

---

## 핵심 개념 7. ACF와 PACF: 과거 몇 시점까지 볼지 정하는 힌트

ARIMA에서 \(p\), \(q\)를 아무렇게나 정하면 안 됩니다. Box-Jenkins 접근법은 ACF와 PACF를 통해 후보를 정합니다.

ACF는 **자기상관함수**입니다.

쉽게 말하면 다음 질문입니다.

**지금 값은 1분 전, 2분 전, 3분 전 값과 얼마나 닮았나?**

예를 들어 온도 데이터에서 lag 1의 ACF가 높다면, 현재 온도는 1분 전 온도와 강하게 관련되어 있다는 뜻입니다.

PACF는 **부분 자기상관함수**입니다.

질문이 조금 다릅니다.

**1분 전의 영향을 제거하고도, 2분 전 값이 현재와 직접 관련이 있나?**

실무적으로는 이렇게 이해하면 됩니다.

| 도구 | 보는 것 | 도움 되는 부분 |
|---|---|---|
| ACF | 과거 값들과의 전체 상관 | MA 후보 차수 \(q\) |
| PACF | 중간 lag 영향을 제거한 직접 상관 | AR 후보 차수 \(p\) |

입문 단계에서는 ACF/PACF를 완벽히 외우기보다, “과거 몇 시점까지 의미 있는 기억이 남아 있는지 보는 도구”라고 이해하면 충분합니다.

## 핵심 개념 8. Box-Jenkins 절차: 모델링은 한 번에 끝나지 않는다

Box-Jenkins 접근법의 가장 중요한 점은 모델 하나를 던지고 끝내지 않는다는 것입니다.

흐름은 다음과 같습니다.

| 단계 | 하는 일 | 설비 데이터 예시 |
|---|---|---|
| 1. 시각화 | 원본 시계열을 본다 | 온도, 압력, 진동 RMS 추세 확인 |
| 2. 정상성 확인 | 추세/계절성/분산 변화 확인 | 부하 증가로 평균이 바뀌는지 확인 |
| 3. 변환/차분 | 정상성에 가깝게 만든다 | 온도 변화량, 로그 변환 등 |
| 4. 모델 식별 | ARIMA 후보를 정한다 | ACF/PACF로 \(p,d,q\) 후보 설정 |
| 5. 추정 | 파라미터를 학습한다 | \(\phi, \theta\) 추정 |
| 6. 진단 | 잔차가 랜덤한지 확인한다 | 잔차에 패턴이 남으면 모델 불충분 |
| 7. 예측 | 미래 값을 예측한다 | 다음 10분 온도 예측 |
| 8. 반복 | 부족하면 다시 수정한다 | 계절성, 외생변수, 다른 모델 검토 |

이 절차가 중요한 이유는 **잔차 진단** 때문입니다. 모델이 좋은지 나쁜지는 예측값만 보고 판단하면 안 됩니다. 모델이 설명하지 못한 잔차에 여전히 패턴이 남아 있으면, 모델이 중요한 구조를 놓친 것입니다.

---

## 핵심 개념 9. 잔차는 이상탐지의 출발점이다

설비 진단 관점에서 ARIMA의 가장 실용적인 사용법은 예측 자체보다 **잔차 기반 이상탐지**입니다.

예를 들어 정상 구간의 온도 패턴으로 모델을 학습했다고 하겠습니다.

| 시간 | 실제 온도 | ARIMA 예측 | 잔차 |
|---|---:|---:|---:|
| 09:00 | 70.0 | 70.1 | -0.1 |
| 09:01 | 70.2 | 70.2 | 0.0 |
| 09:02 | 70.4 | 70.3 | +0.1 |
| 09:03 | 70.5 | 70.5 | 0.0 |
| 09:04 | 76.8 | 70.6 | +6.2 |

09:04의 실제 온도는 모델 예측보다 6.2도 높습니다. 이 잔차가 평소보다 크다면 이상 후보입니다.

이 접근은 설비 데이터에서 자주 씁니다.

- 정상 데이터로 예측 모델을 학습한다.
- 새로운 센서값이 들어오면 정상 패턴을 기준으로 예측한다.
- 실제값과 예측값의 차이를 본다.
- 잔차가 임계값보다 크면 이상으로 본다.

다만 여기서도 주의해야 합니다. 부하, 운전 모드, 외기 온도 같은 조건이 바뀌면 잔차가 커질 수 있습니다. 그게 진짜 고장인지, 운전 조건 변화인지는 별도로 구분해야 합니다.

---

## 작은 예시: 온도 센서 1분 단위 데이터로 ARIMA 감 잡기

가상의 온도 센서 데이터입니다.

| 시각 | 온도 |
|---|---:|
| 09:00 | 70.0 |
| 09:01 | 70.2 |
| 09:02 | 70.4 |
| 09:03 | 70.7 |
| 09:04 | 70.9 |
| 09:05 | 71.1 |
| 09:06 | 71.4 |
| 09:07 | 71.5 |

이 데이터는 조금씩 증가합니다. 원본값만 보면 추세가 있습니다.

1차 차분을 해보겠습니다.

| 시각 | 온도 | 1차 차분 |
|---|---:|---:|
| 09:00 | 70.0 | - |
| 09:01 | 70.2 | 0.2 |
| 09:02 | 70.4 | 0.2 |
| 09:03 | 70.7 | 0.3 |
| 09:04 | 70.9 | 0.2 |
| 09:05 | 71.1 | 0.2 |
| 09:06 | 71.4 | 0.3 |
| 09:07 | 71.5 | 0.1 |

차분값은 대략 0.1~0.3 사이에서 움직입니다. 원본보다 훨씬 안정적입니다.

이제 모델은 이런 질문을 합니다.

- 온도 변화량은 직전 변화량과 관련이 있는가?
- 최근 예측 오차가 다음 변화량에 영향을 주는가?
- 잔차에 아직 패턴이 남아 있는가?

이 과정을 통해 “정상 온도 패턴”을 모델링할 수 있습니다.

## 설비 진단 예시: sliding window를 만들 때의 위험

시계열을 ML 모델에 넣으려면 보통 window로 자릅니다.

예를 들어 window 길이를 5분으로 두고 다음 1분 온도를 예측한다고 하겠습니다.

| window 입력 | 예측 target |
|---|---|
| 09:00~09:04 온도 | 09:05 온도 |
| 09:01~09:05 온도 | 09:06 온도 |
| 09:02~09:06 온도 | 09:07 온도 |

여기서 실무자가 자주 하는 실수가 있습니다.

**이 window들을 랜덤하게 train/test로 나누는 것**입니다.

예를 들어 다음 두 window를 보세요.

| window | 포함 시간 |
|---|---|
| A | 09:00~09:04 |
| B | 09:01~09:05 |

A와 B는 거의 같은 데이터를 공유합니다. 그런데 A가 train에 들어가고 B가 test에 들어가면 어떻게 될까요?

모델은 사실상 test와 매우 비슷한 데이터를 이미 본 것입니다. 이것이 시계열 데이터 누수의 대표적인 형태입니다.

설비 진단에서는 더 위험합니다.

- 같은 베어링의 같은 고장 run에서 잘라낸 window가 train/test에 섞인다.
- 같은 모터의 인접 운전 구간이 train/test에 섞인다.
- 같은 설비의 같은 부하 조건 데이터가 양쪽에 들어간다.
- 모델은 고장 일반화가 아니라 특정 설비의 신호 fingerprint를 외운다.

그래서 설비 진단에서는 가능하면 다음과 같은 분할을 써야 합니다.

| 상황 | 권장 split |
|---|---|
| 여러 설비가 있음 | 설비 unit 단위 split |
| 여러 run이 있음 | run 단위 split |
| 하나의 긴 run만 있음 | 시간 순서 기반 split |
| 여러 운전 조건이 있음 | 조건별 holdout도 별도 평가 |
| RUL 데이터 | engine/bearing unit 단위 split |

예를 들어 bearing fault diagnosis라면 다음이 더 현실적입니다.

| train | test |
|---|---|
| Bearing 1, Bearing 2, Bearing 3 | Bearing 4 |
| Run 1~7 | Run 8~10 |
| 0hp, 1hp 조건 | 2hp 조건 holdout |

반대로 아래는 위험합니다.

| train | test |
|---|---|
| 전체 window 중 랜덤 80% | 전체 window 중 랜덤 20% |

이렇게 하면 accuracy가 99%가 나와도 실무 성능이라고 믿기 어렵습니다.

## 실무에서 어떻게 써야 하나?

ARIMA/Box-Jenkins는 최신 딥러닝보다 단순하지만, 실무에서는 여전히 유용합니다.

첫째, **baseline**으로 씁니다.  
딥러닝 모델을 만들기 전에 ARIMA, seasonal naive, moving average 같은 단순 모델을 반드시 돌려봐야 합니다. 딥러닝이 이 단순 baseline을 이기지 못하면 복잡한 모델을 쓸 이유가 없습니다.

둘째, **센서별 정상 패턴 예측**에 씁니다.  
온도, 압력, 유량, 전류 RMS처럼 비교적 저주파로 움직이는 센서에는 ARIMA 계열 모델이 좋은 출발점이 됩니다.

셋째, **잔차 기반 이상탐지**에 씁니다.  
정상 데이터로 모델을 만들고, 예측값과 실제값의 차이가 커지는 시점을 이상 후보로 볼 수 있습니다.

넷째, **데이터 품질 점검**에 씁니다.  
갑작스러운 spike, sensor dropout, drift, calibration issue를 잔차나 차분값으로 빠르게 확인할 수 있습니다.

다섯째, **딥러닝 모델의 성능을 의심하는 기준**으로 씁니다.  
설비 진단 딥러닝 논문에서 accuracy가 너무 높게 나오면 먼저 split을 확인해야 합니다. random window split이라면, ARIMA보다 성능이 높아도 실제 현장 일반화 성능은 낮을 수 있습니다.

## 설비 센서별로 보면 어디에 잘 맞나?

| 센서 유형 | ARIMA 적용 가능성 | 설명 |
|---|---|---|
| 온도 | 높음 | 천천히 변하고 자기상관이 강함 |
| 압력 | 중간~높음 | 운전 조건이 안정적이면 예측 가능 |
| 전류 RMS | 중간 | 부하 변화의 영향을 많이 받음 |
| 진동 RMS | 중간 | 저주파 요약값이면 가능 |
| raw vibration waveform | 낮음 | 고주파 신호라 FFT, envelope, wavelet, CNN이 더 적합한 경우 많음 |
| RUL degradation signal | 중간 | 단순 예측보다는 degradation modeling과 결합 필요 |

고주파 진동 데이터는 ARIMA만으로는 부족한 경우가 많습니다. 예를 들어 베어링 결함은 특정 주파수 대역에서 나타날 수 있으므로 FFT, envelope analysis, wavelet transform을 함께 봐야 합니다. 하지만 진동 RMS, kurtosis, crest factor 같은 feature를 시간에 따라 추적하는 경우에는 ARIMA식 사고가 도움이 됩니다.

## 한계

첫째, ARIMA는 기본적으로 **선형 모델**입니다.  
설비 데이터는 부하, 온도, 회전수, 마모, 윤활 상태가 얽혀 비선형적으로 움직일 수 있습니다.

둘째, 정상성 가정이 부담스럽습니다.  
현장 설비는 운전 조건이 바뀌고, 센서 drift가 생기고, 정비 후 패턴이 바뀝니다. 차분만으로 해결되지 않는 경우가 많습니다.

셋째, 다변량 센서 관계를 다루기 어렵습니다.  
ARIMA는 주로 단변량 시계열에서 출발합니다. 여러 센서 간 상호작용을 다루려면 VAR, state-space model, deep learning, graph model 등이 필요할 수 있습니다.

넷째, 고주파 진동 raw signal에는 직접 적용하기 어렵습니다.  
베어링 결함 진단에서는 시간축 waveform 자체보다 주파수 영역의 결함 주파수, sideband, envelope spectrum이 더 중요할 수 있습니다.

다섯째, 이상탐지 임계값 설정이 어렵습니다.  
잔차가 커졌다고 무조건 고장은 아닙니다. 운전 조건 변화, 센서 교체, calibration 문제, 부하 변화도 잔차를 키울 수 있습니다.

여섯째, split을 잘못하면 평가가 무너집니다.  
ARIMA든 딥러닝이든 window 단위 랜덤 split은 위험합니다. 같은 설비의 인접 window가 train/test에 섞이면 모델 성능이 과대평가됩니다.

## 오늘 공부용 요약

오늘의 핵심은 ARIMA 자체보다 **시계열을 바라보는 기본 태도**입니다.

1. 시계열은 행을 섞으면 안 된다.
2. 현재 값은 과거 값과 연결되어 있다.
3. 추세가 있으면 차분해서 변화량을 볼 수 있다.
4. AR은 과거 값, MA는 과거 오차를 이용한다.
5. ARIMA는 차분 + AR + MA를 합친 모델이다.
6. 모델을 만들고 끝내는 것이 아니라 잔차를 반드시 확인해야 한다.
7. 잔차는 이상탐지의 출발점이 될 수 있다.
8. 설비 진단에서는 window 단위 랜덤 split을 조심해야 한다.
9. accuracy만 보지 말고 false alarm, missed detection, recall, F1, lead time을 봐야 한다.
10. 딥러닝을 쓰더라도 ARIMA 같은 단순 baseline을 먼저 이겨야 한다.
