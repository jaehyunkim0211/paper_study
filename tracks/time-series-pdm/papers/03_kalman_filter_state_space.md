# 3편: A New Approach to Linear Filtering and Prediction Problems

## 결론부터

**이 논문의 핵심 결론은, 노이즈가 섞인 센서 관측값만 보지 말고 “설비의 보이지 않는 진짜 상태”를 상태공간모델로 표현한 뒤, 예측과 보정을 반복하면서 최적의 상태 추정치를 갱신할 수 있다는 것입니다.**

## 이 논문을 한 문장으로 요약하면

**Kalman Filter는 “모델이 예상한 값”과 “센서가 실제로 측정한 값”을 불확실성에 따라 적절히 섞어서, 매 시점마다 숨은 상태를 재귀적으로 추정하는 필터링 방법입니다.**

원 논문은 기존 Wiener filtering/prediction 문제를 상태전이 관점으로 다시 정리했고, 최적 추정 오차의 covariance matrix에 대한 차분방정식을 구하면 최적 선형 필터의 계수를 얻을 수 있음을 보였습니다.

## 왜 이 논문이 필요했나?

1편 ARIMA는 과거 값과 오차로 미래를 예측하는 법을, 2편 STL은 추세·계절성·잔차를 나누는 법을 알려줬습니다. 그런데 실무 설비 데이터에는 한 가지 문제가 더 있습니다. **우리가 정말 알고 싶은 설비의 상태는 직접 보이지 않습니다.**

| 우리가 알고 싶은 것 | 실제로 센서가 주는 것 |
|---|---|
| 베어링 마모 정도 | 진동 RMS, envelope, 온도 |
| 윤활 상태 | 온도, 전류, 소음 |
| 모터 부하 상태 | 전류, 토크 추정값 |
| 펌프 내부 이상 | 압력, 유량, 전류 |
| 설비 열화 단계 | 여러 센서의 간접 신호 |

센서값만 믿으면 너무 흔들리고, 모델만 믿으면 실제 이상을 놓칠 수 있습니다. Kalman Filter의 핵심은 이 둘을 균형 있게 섞는 것입니다.

> “내 모델은 설비 상태가 이 정도일 것이라고 예상한다. 그런데 센서는 조금 다른 값을 말한다. 모델과 센서 중 무엇을 얼마나 믿을까?”

## 핵심 개념 1. 상태공간모델: 보이지 않는 상태와 보이는 관측을 나눈다

Kalman Filter를 이해하려면 먼저 **상태(state)**와 **관측(observation)**을 나눠야 합니다.

| 구분 | 자동차 예시 | 설비 예시 |
|---|---|---|
| 상태 | 실제 위치와 속도 | 실제 마모 정도, 실제 열 상태 |
| 관측 | GPS 위치 | 진동, 온도, 전류, 압력 |
| 문제 | GPS가 튐 | 센서가 noisy하고 상태는 직접 보이지 않음 |

상태공간모델은 복잡한 신호를 “상태가 시간에 따라 어떻게 변하고, 관측값이 상태를 어떻게 반영하는가”라는 구조로 봅니다.

## 핵심 개념 2. 상태방정식

상태방정식은 이렇게 묻습니다.

**지금 설비 상태를 알면, 다음 시점의 설비 상태를 어느 정도 예측할 수 있나?**

\[
x_t = F x_{t-1} + w_t
\]

| 기호 | 의미 | 설비 예시 |
|---|---|---|
| \(x_t\) | 현재의 숨은 상태 | 실제 온도, 실제 마모 정도 |
| \(F\) | 상태가 다음 시점으로 넘어가는 방식 | 온도가 천천히 이어지는 정도 |
| \(w_t\) | process noise | 부하 급변, 냉각 변화, 마찰 변화 |

## 핵심 개념 3. 관측방정식

관측방정식은 이렇게 묻습니다.

**센서값은 숨은 상태를 어떻게 반영하는가?**

\[
z_t = H x_t + v_t
\]

| 기호 | 의미 | 설비 예시 |
|---|---|---|
| \(z_t\) | 관측값 | 센서가 측정한 온도, 진동, 전류 |
| \(H\) | 상태가 관측값으로 나타나는 방식 | 실제 온도가 센서 온도로 반영되는 관계 |
| \(v_t\) | measurement noise | 센서 오차, 전기적 노이즈, 통신 잡음 |

## 핵심 개념 4. 예측 → 보정 반복

Kalman Filter는 매우 단순한 리듬을 가집니다.

| 단계 | 질문 | 설비 예시 |
|---|---|---|
| 예측 prediction | 모델상 지금쯤 상태가 얼마일까? | 모터 온도가 70.5도쯤일 것 |
| 관측 observation | 센서는 뭐라고 말하나? | 센서가 71.2도를 측정 |
| 보정 update | 모델과 센서를 어떻게 섞을까? | 모델도 믿고 센서도 일부 반영 |
| 반복 recursion | 다음 시점으로 넘어간다 | 1분 뒤 다시 예측하고 보정 |

## 핵심 개념 5. Kalman Gain

Kalman Gain은 다음 질문의 답입니다.

**센서값을 얼마나 반영할 것인가?**

| 상황 | Kalman Gain | 의미 |
|---|---:|---|
| 센서가 매우 정확함 | 큼 | 센서값을 많이 믿음 |
| 센서가 매우 noisy함 | 작음 | 모델 예측을 더 믿음 |
| 모델이 불확실함 | 큼 | 관측값으로 많이 보정 |
| 모델이 매우 확실함 | 작음 | 관측값을 조금만 반영 |

\[
K_t = P_{t|t-1}H^T(H P_{t|t-1}H^T + R)^{-1}
\]

핵심은 단순합니다. \(P\)가 크면 모델 예측이 불확실하므로 센서를 더 믿고, \(R\)이 크면 센서가 noisy하므로 센서를 덜 믿습니다.

## 핵심 개념 6. Covariance \(P\)

Kalman Filter는 상태값만 추정하지 않습니다. **그 추정이 얼마나 불확실한지도 같이 추적합니다.**

| 상황 | \(P\) 해석 |
|---|---|
| 센서가 오래 끊김 | 상태 추정 불확실성 증가 |
| 운전 조건이 갑자기 바뀜 | 모델 예측 불확실성 증가 |
| 안정적으로 측정값이 들어옴 | 추정 불확실성 감소 |
| 센서 drift 의심 | measurement noise 또는 bias 고려 필요 |

## 핵심 개념 7. Innovation: 이상탐지 신호

\[
\text{innovation}_t = z_t - H\hat{x}_{t|t-1}
\]

쉽게 말하면 **센서가 실제로 말한 값 - 모델이 예상한 센서값**입니다.

| 시간 | 예측 온도 | 센서 온도 | innovation |
|---|---:|---:|---:|
| 09:00 | 70.0 | 70.1 | +0.1 |
| 09:01 | 70.2 | 70.3 | +0.1 |
| 09:02 | 70.4 | 70.2 | -0.2 |
| 09:03 | 70.5 | 76.0 | +5.5 |

1편 ARIMA에서는 예측 잔차, 2편 STL에서는 remainder, 3편 Kalman Filter에서는 innovation을 봅니다. 셋의 철학은 같습니다. **정상 패턴으로 설명되지 않는 차이가 커지면 이상 후보로 본다.**

## 핵심 개념 8. Filtering, Prediction, Smoothing

| 용어 | 사용하는 정보 | 실무 의미 |
|---|---|---|
| filtering | 현재까지의 관측값 | 실시간 모니터링 |
| prediction | 현재까지의 정보로 미래 예측 | 예측 경보, RUL 예측 |
| smoothing | 과거와 미래 관측값 모두 사용 | 사후 분석 |

실시간 성능 평가에서 smoothing 결과를 쓰면 미래 정보를 사용한 것이므로 성능이 과대평가될 수 있습니다.

## 작은 예시 데이터

| 시간 | 센서 측정 온도 |
|---|---:|
| 09:00 | 70.4 |
| 09:01 | 70.8 |
| 09:02 | 71.0 |
| 09:03 | 76.0 |
| 09:04 | 71.3 |
| 09:05 | 71.5 |

09:03의 76.0은 실제 과열일 수도 있고 센서 spike일 수도 있습니다. Kalman Filter는 모델 예측, 센서값, Kalman Gain을 이용해 보정합니다.

| 시간 | 모델 예측 | 센서값 | Kalman Gain | 보정 후 추정 | innovation |
|---|---:|---:|---:|---:|---:|
| 09:00 | 70.0 | 70.4 | 0.60 | 70.24 | +0.40 |
| 09:01 | 70.3 | 70.8 | 0.55 | 70.58 | +0.50 |
| 09:02 | 70.7 | 71.0 | 0.50 | 70.85 | +0.30 |
| 09:03 | 71.0 | 76.0 | 0.20 | 72.00 | +5.00 |
| 09:04 | 72.1 | 71.3 | 0.45 | 71.74 | -0.80 |

09:03에서 필터는 센서값 76.0을 전부 믿지는 않지만, innovation +5.00은 이상 후보로 기록해야 합니다.

## 실무에서 어떻게 써야 하나?

### 1. 센서 smoothing이 아니라 상태 추정으로 본다

Kalman Filter는 단순 이동평균이 아닙니다. “설비의 숨은 상태가 시간에 따라 어떻게 변하고, 센서가 그 상태를 어떻게 noisy하게 관측하는가?”를 모델링합니다.

### 2. \(Q\)와 \(R\)을 현장 지식으로 조정한다

| 파라미터 | 의미 | 너무 작게 잡으면 | 너무 크게 잡으면 |
|---|---|---|---|
| \(Q\) | 상태가 실제로 얼마나 흔들리는가 | 실제 변화를 못 따라감 | 상태 추정이 너무 출렁임 |
| \(R\) | 센서 측정 noise가 얼마나 큰가 | 센서 spike에 과민 반응 | 실제 이상을 늦게 반영 |

### 3. Innovation을 anomaly score로 쓴다

innovation이 한 번만 튄 것인지, 여러 시점 지속되는지, 여러 센서에서 동시에 나타나는지, 특정 운전 조건에서만 나타나는지 확인해야 합니다.

### 4. RUL에서는 degradation state를 추적한다

\[
x_t =
\begin{bmatrix}
\text{degradation level} \\
\text{degradation rate}
\end{bmatrix}
\]

RUL 평가는 accuracy가 아니라 **MAE, RMSE, NASA scoring function, early/late prediction penalty**를 봐야 합니다.

## 데이터 분할에서 특히 조심할 점

Kalman Filter를 쓰더라도 데이터 분할 원칙은 바뀌지 않습니다. **시계열/설비 진단에서는 모델보다 데이터 분할이 먼저입니다.**

| 위험한 방식 | 왜 위험한가 |
|---|---|
| 전체 데이터에 Kalman smoothing 적용 후 feature 생성 | 미래 test 정보가 과거 feature에 섞일 수 있음 |
| sliding window를 만든 뒤 랜덤 train/test split | 인접 window가 양쪽에 섞임 |
| 같은 설비 run에서 나온 window가 train/test에 동시에 포함 | 모델이 설비 fingerprint를 외움 |

## 한계

1. 기본 Kalman Filter는 선형 가정이 강합니다.
2. Gaussian noise 가정이 항상 맞지 않습니다.
3. \(Q\), \(R\) 튜닝이 어렵습니다.
4. 상태를 잘못 정의하면 결과가 의미 없습니다.
5. smoothing을 평가에 쓰면 data leakage가 생길 수 있습니다.

## 이전 논문들과 연결해서 이해하기

| 관점 | 1편 ARIMA | 2편 STL | 3편 Kalman Filter |
|---|---|---|---|
| 핵심 질문 | 과거 값/오차로 현재를 설명할 수 있나? | 현재 값은 추세, 반복 패턴, 잔차 중 무엇 때문인가? | 노이즈 관측으로 숨은 상태를 추정할 수 있나? |
| 주요 산출물 | 예측값, 잔차 | trend, seasonal, remainder | state estimate, covariance, innovation |
| 이상탐지 연결 | 예측 잔차 | remainder | innovation |
| 누수 주의점 | window random split | 전체 STL 후 split | smoothing 후 split |

## 오늘 공부용 요약

1. Kalman Filter는 노이즈가 섞인 센서값으로 보이지 않는 진짜 상태를 추정합니다.
2. 상태방정식은 설비 상태가 시간에 따라 어떻게 변하는지 설명합니다.
3. 관측방정식은 센서값이 숨은 상태를 어떻게 반영하는지 설명합니다.
4. Kalman Gain은 모델 예측과 센서 관측 중 무엇을 얼마나 믿을지 결정합니다.
5. Innovation은 이상탐지 score로 쓸 수 있습니다.
6. Filtering은 현재까지의 정보만 쓰고, smoothing은 미래 정보도 쓸 수 있으므로 평가에서 leakage를 조심해야 합니다.
7. 설비 진단에서는 sliding window random split을 피하고 unit/run/time 기준 split을 우선해야 합니다.
